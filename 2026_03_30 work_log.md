# 2026_03_30 work_log

## 문제점
- 금요일에 서버에 있는 파일들의 경로를 수정함
- docker-compose.volumes.yml에서 경로를 이전 경로를 계속 바라보고 있어 python파일이 제대로 실행되지 않는 문제점이 발생함
  1. slackwebhook.py를 불러오는데 slackwebhook의 경로를 서버 경로로 보고 있어 제대로 실행되지 않았을 수도..
  2. 기존에는 airflow standalone을 사용하고 있는데 `airflow standalone`은 로컬에서 Airflow를 빨리 띄워서 테스트하거나 학습할 때 쓰는 명령어라서 실제 운영에는 적합하지 않음..
- db를 초기화해도 volumes.yml이 예전 경로를 타고 있었음
- airflow standalone은 로컬에서 test용이지 실제 운영에서는 사용하지 않음.. (나는 airflow standalone을 사용하고 있었음)


## 해결책
1. Slackwebhook을 docker 경로에 맞추기
```
from pathlib import Path
import sys

DAGS_DIR = Path(__file__).resolve().parent
if str(DAGS_DIR) not in sys.path:
    sys.path.append(str(DAGS_DIR))
```
-> 현재 파일 위치를 기준으로 파일이 들어있는 폴더 경로를 `DAGS_DIR`에 저장
예를 들어, 현재 파일이 `/opt/airflow/dags/test_dag.py` 라면, `DAGS_DIR`는 `opt/airflow/dags`임.

또한, 다른 파일에서 load_dotenv(".env")를 했더니 제대로 찾지 못하는 문제점 발생 
-> `load_dotenv(Path(__file__).resolve().parent / ".env")` 로 작성하여 서버 경로가 아닌 docker 경로로 지정함
만약 현재 `file_path = "/home/linuxuser/wemarketing/Data_collection/Crawling/benow/test.py"` 이라면 Path(__file__).resolve().parent는 `/home/linuxuser/wemarketing/Data_collection/Crawling/benow` 임.

✴️ `Path(__file__).resolve().parent[0]`과 `Path(__file__).resolve().parent[1]`의 차이점
- 우선 parent[0]은, 현재 내 경로가 `file_path`라면, `/home/linuxuser/wemarketing/Data_collection/Crawling/benow/`이고
- parent[1]은, `/home/linuxuser/wemarketing/Data_collection/Crawling/` 을 의미함.

2. `docker-compose.yml`이랑 `docker-compose.volumes.yml`의 차이
- docker-compose.yml = Airflow 자체를 어떻게 띄울지
- docker-compose.volumes.yml = Airflow 컨테이너들이 어떤 폴더를 볼지..
  - `/home/linuxuser/wemarketing : /opt/airflow/app` 라고 지정했기 때문에 경로를 정함
  - AIRFLOW__CORE__DAGS_FOLDER: /opt/airflow/dags 는 dag 경로를 설정하는 코드
 
3. 웹 Trigger 와 CLI Trigger의 차이
- 오히려 0.02s 만에 끝나는 에러는 제대로 dag가 경로를 못 찾아갔다고 생각. 그래서 ⭐`docker compose -f docker-compose.yml -f docker-compose.volumes.yml exec airflow-apiserver airflow dags trigger <DAG_ID>`⭐ 로 바로 실행되는지 테스트 가능

4. standalone의 문제점
- 기존에 airflow standalone을 사용함. 하지만 이는 로컬에서 airflow가 정상적으로 작동하는지 체크하기 위함일 뿐, 실제 서비스 환경에서는 standalone을 사용하면 안 됨. -> ⭐ **scheduler, webserver, triggerer, DB 설정 등을 나눠서 관리하는 건 조금 더 제대로 된 구성**

5.  &와 *의 차이
- `&airflow-common-env` - 공통 환경변수를 한 번 정의해주고, 다른 서비스에서 `airflow-common-env` 로 재사용하는 방식
- airflow-apiserver
- airflow-scheduler
- airflow-dag-processor
- airflow-triggerer
이 서비스들이 전부 같은 Airflow 설정을 공유하게 만듦.

### 이외
docker.desktop을 설치하여 회의실 프로젝트를 위한 이미지를 docker hub에 push하였음

1. 이미지 이름을 Docker Hub 형식으로 만들기
예)
`docker build -t wemadev/meeting_room_bot:v1 . `
2. 이미지 확인

`docker images`

3. Docker Hub로 업로드

`docker push wemadev/meeting_room_bot:v1`
ℹ️ docker는 git과 다르게 push할 때마다 tag를 바꿔주지 않아도 됨.

4. 만약 pull을 하고 싶다면
```
docker build -t wemadev/meeting_room_bot:latest .
docker push wemadev/meeting_room_bot:latest
```
하면 됨.
