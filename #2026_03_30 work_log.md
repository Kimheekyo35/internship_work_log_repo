## 문제점
- 금요일에 서버에 있는 파일들의 경로를 수정함
- docker-compose.volumes.yml에서 경로를 이전 경로를 계속 바라보고 있어 python파일이 제대로 실행되지 않는 문제점이 발생함
  1. slackwebhook.py를 불러오는데 slackwebhook의 경로를 서버 경로로 보고 있어 제대로 실행되지 않았을 수도..
  2. 기존에는 airflow standalone을 사용하고 있는데 `airflow standalone`은 로컬에서 Airflow를 빨리 띄워서 테스트하거나 학습할 때 쓰는 명령어라서 실제 운영에는 적합하지 않음..
- db를 초기화해도 volumes.yml이 예전 경로를 타고 있었음
- airflow standalone은 로컬에서 test용이지 실제 운영에서는 사용하지 않음.. (나는 airflow standalone을 사용하고 있었음)


## 해결책
1. Slackwebhook을 docker 경로에 맞추기
```
from pathlib import Path
import sys

DAGS_DIR = Path(__file__).resolve().parent
if str(DAGS_DIR) not in sys.path:
    sys.path.append(str(DAGS_DIR))
```
-> 현재 파일 위치를 기준으로 파일이 들어있는 폴더 경로를 `DAGS_DIR`에 저장
예를 들어, 현재 파일이 `/opt/airflow/dags/test_dag.py` 라면, `DAGS_DIR`는 `opt/airflow/dags`임.

또한, 다른 파일에서 load_dotenv(".env")를 했더니 제대로 찾지 못하는 문제점 발생 
-> `load_dotenv(Path(__file__).resolve().parent / ".env")` 로 작성하여 서버 경로가 아닌 docker 경로로 지정함
만약 현재 `file_path = "/home/linuxuser/wemarketing/Data_collection/Crawling/benow/test.py"` 이라면 Path(__file__).resolve().parent는 `/home/linuxuser/wemarketing/Data_collection/Crawling/benow` 임.

2. `docker-compose.yml`이랑 `docker-compose.volumes.yml`의 차이
- docker-compose.yml = Airflow 자체를 어떻게 띄울지
- docker-compose.volumes.yml = Airflow 컨테이너들이 어떤 폴더를 볼지..
  - `/home/linuxuser/wemarketing : /opt/airflow/app` 라고 지정했기 때문에 경로를 정함
  - AIRFLOW__CORE__DAGS_FOLDER: /opt/airflow/dags 는 dag 경로를 설정하는 코드
